# 01\_ROS2基础\.md

## 1\. ROS2是什么（通俗理解）

ROS2是机器人的**操作系统级开发框架**，不是真正的系统，是运行在Ubuntu上的开发工具集。核心作用是**解耦机器人各个功能模块**：传感器、控制、建图、导航、可视化可以独立开发，通过统一的话题msg、服务srv、参数通信param。实际上就是基于dds的分布式通信框架，主要用于各个模块之间的通信

对比ROS1：ROS2支持实时性、多线程、跨平台，是目前工业机器人、移动机器人、自动驾驶的主流开发框架。

### 2\.1 节点（Node）

机器人每一个功能都是一个节点：雷达驱动节点、相机节点、控制器节点、SLAM节点、导航节点。我们的项目中，大部分节点由ROS2官方包启动，无需自己写源码，属于**配置型工程**。

### 2\.2 话题（Topic）msg格式

节点之间传输数据的通道，单向发布\-订阅模式。本项目核心话题：

- `/scan`：激光雷达数据（SLAM、导航核心输入）

- `/camera/image_raw`：相机图像数据

- `/cmd_vel`：机器人速度指令（导航输出、手动控制输出）

- `/diff_drive_controller/odom`：里程计数据

### 2\.3 TF坐标变换

机器人所有部件、传感器都有独立坐标系，TF负责告诉机器人“雷达在底盘哪个位置、底盘在地图哪个位置”，是**建图、定位、导航的基础**。

### 2\.4 参数（Parameter）

所有控制器、SLAM、Nav2的配置都以参数文件（yaml）形式存在，无需改代码，改参数即可调优机器人性能，这是工程化开发的核心。

## 3\. 常用基础命令（项目高频使用）

# 02\_Workspace与Package\.md

## 1\. ROS2工作空间（Workspace）通俗讲解

工作空间是存放所有机器人源码工程的**总文件夹**，默认命名 `ros2_ws`。工作空间核心结构：

- `src`：源码目录，存放所有功能包（我们的 `my_robot_description` 在此处）

- `build`：编译缓存目录

- `install`：编译安装目录（最终运行的代码/配置都在这里）

- `log`：日志目录

## 2\. 核心原理：src与install的关系（重点巩固）

**src是源码仓库，install是运行仓库**。

我们修改的 urdf、launch、yaml 配置文件都在 src 目录，但是ROS2运行时不会直接读取src文件，必须执行 `colcon build`，将文件**拷贝、安装**到 `install/share/功能包名` 目录。

这就是为什么修改配置、模型、启动文件后，必须重新编译才能生效的核心原因。

## 3\. 功能包（Package）概念

功能包是ROS2的最小工程单元，一个功能包对应一套独立功能。本项目唯一功能包 `my_robot_description`，集成了机器人建模、仿真、控制、建图、导航所有配置。

## 4\. 自定义功能包核心目录作用（对应项目结构）

- `launch/`：启动文件，一键启动所有节点

- `urdf/`：机器人模型文件

- `config/`：所有yaml参数配置

- `maps/`：保存的地图文件

- `rviz/`：RViz保存的配置

- `worlds/`：Gazebo仿真环境场景

## 5\. 工作空间常用操作命令

```Plain Text
# 进入工作空间
cd ~/ros2_ws
# 编译全部包
colcon build
# 编译单个包
colcon build --packages-select my_robot_description
# 刷新环境变量
source install/setup.bash
```

# 03\_CMakeLists与package\.md

## 1\. package\.xml 功能包清单

`package.xml` 是功能包的“身份证”，告诉ROS2当前包的名称、版本、依赖库、运行权限。

### 1\.1 核心字段作用

- `<name>`：功能包名，必须和文件夹名一致

- `<depend>`：声明依赖包（gazebo、ros2\_control、nav2、slam\_toolbox等）

- `<buildtool_depend>ament_cmake`：声明编译类型为cmake

### 1\.2 项目巩固要点

所有launch、模型、配置依赖的官方包，都必须在package\.xml中声明，否则编译无报错，运行时会出现**节点找不到、资源加载失败**的隐性BUG。

## 2\. CMakeLists\.txt 编译配置文件

负责定义编译规则，本项目核心作用：**将资源文件安装到install目录**。

### 2\.1 核心指令讲解

```Plain Text
install(DIRECTORY launch urdf config rviz maps worlds
  DESTINATION share/${PROJECT_NAME}
)
```

这段代码的作用：将项目中的所有资源文件夹，编译后拷贝到install目录，保证Launch文件可以通过 `FindPackageShare` 正常读取资源。

## 3\. 两个文件的对应关系

package\.xml管**依赖声明**，CMakeLists管**文件安装与编译**，二者缺一不可，是ROS2功能包正常运行的基础。

# 04\_Launch系统\.md

## 1\. Launch是什么

Launch是ROS2的**批量启动工具**，可以一键启动多个节点、加载参数、加载模型、启动仿真、设置延时、关联节点启动顺序，替代手动一个个敲命令。

## 2\. 本项目三大核心Launch文件详解

### 2\.1 display\.launch\.py（全功能启动文件）

项目最核心启动文件，一次性启动全套仿真环境：Gazebo仿真、RViz可视化、机器人模型解析、机器人状态发布、ros2\_control控制器、cmd\_vel话题转发、SLAM建图节点。

启动流程：解析Xacro模型 → 生成robot\_description → 启动robot\_state\_publisher发布TF → 启动Gazebo环境 → 生成机器人实体 → 加载控制器 → 启动SLAM → 启动RViz。

### 2\.2 slam\.launch\.py（轻量化建图启动）

单独启动slam\_toolbox节点，用于只需要建图、不需要完整仿真的场景，轻量化、启动更快。

### 2\.3 navigation\.launch\.py（导航启动文件）

加载保存好的地图，启动Nav2全套导航节点、AMCL定位、话题转发、RViz导航配置，实现自主导航。

## 3\. Launch核心语法与原理

- `Command(["xacro"])`：解析xacro文件，生成机器人模型参数 `robot_description`

- `robot_state_publisher`：读取模型，发布所有静态、动态TF坐标

- `spawn_entity.py`：在Gazebo中生成机器人实体

- `TimerAction`：延时启动，解决控制器启动过快、Gazebo未初始化的报错

## 4\. 常见Launch报错原因

资源路径错误、未编译更新文件、节点启动顺序错乱、依赖缺失、参数文件路径不对。

# 05\_URDF与Xacro\.md

## 1\. URDF与Xacro区别

URDF是机器人基础模型描述语言，语法繁琐、重复代码多；Xacro是URDF的**增强工具**，支持宏定义、变量、文件拆分、代码复用，是实际机器人建模的标准方案。

## 2\. 本项目Xacro模块化拆分逻辑（核心重点）

项目将机器人拆分为多个独立xacro文件，实现高复用、易修改，各司其职：

- **base\_core\.xacro**：总入口文件，统一引入所有模型、插件、参数

- **robot\_properties\.xacro**：统一管理机器人尺寸、轮半径、轮距、质量、惯量等全局参数

- **base\_body\.xacro**：定义底盘主体、核心坐标系（base\_footprint、base\_link）

- **drive\_wheel\.xacro**：宏定义左右驱动轮，连续运动关节

- **caster\_wheel\.xacro**：支撑万向轮，固定关节

- **sensor\_lidar\.xacro**：激光雷达模型与坐标挂载

- **sensor\_camera\.xacro**：相机模型与光学坐标系

- **gazebo\_plugin\.xacro**：仿真插件、控制插件、传感器数据输出插件

- **materials\.xacro**：模型颜色材质定义

## 3\. Link与Joint核心原理

### 3\.1 Link（刚体）

机器人每一个零部件都是Link，包含三个核心属性：

- visual：RViz显示外观

- collision：仿真碰撞检测

- inertial：物理仿真质量、惯量，影响运动效果

### 3\.2 Joint（关节）

- fixed：固定关节，传感器、支撑轮使用

- continuous：连续旋转关节，差速驱动轮专用

## 4\. 关键坐标系区分（面试高频）

- **base\_footprint**：机器人地面投影坐标系，导航专用，无高度

- **base\_link**：机器人底盘本体坐标系，控制、TF基准

# 06\_TF坐标系\.md

## 1\. TF核心作用

TF是机器人的**空间坐标翻译官**，负责维护所有传感器、底盘、地图、里程计的坐标关系，是SLAM、导航、避障的核心基础。TF断开/错乱，机器人直接无法建图、无法导航。

## 2\. 本项目完整TF树（必背）

map → odom → base\_footprint → base\_link → upper\_base\_link → lidar\_link / camera\_link

## 3\. 各坐标系功能详解

- **map**：全局地图坐标系，绝对静止，导航目标点基于此坐标系

- **odom**：里程计坐标系，随机器人运动累积误差

- **base\_footprint**：机器人地面基准，Nav2导航默认基坐标系

- **base\_link**：机器人本体坐标系，控制、关节基准

- **lidar\_link/camera\_link**：传感器坐标系，决定数据采集位置

## 4\. 项目TF数据流

ros2\_control发布 odom→base\_footprint TF；SLAM发布 map→odom TF；robot\_state\_publisher发布所有静态传感器TF。

## 5\. TF调试命令

```Plain Text
ros2 run tf2_ros tf2_echo map base_footprint
ros2 run tf2_tools view_frames
```

# 08\_ros2\_control\.md

## 1\. ros2\_control是什么

ROS2官方通用机器人控制框架，统一管理所有执行器（轮子、机械臂），实现硬件与上层解耦，仿真和实车可以复用一套控制逻辑。

## 2\. 本项目核心控制器配置（controllers\.yaml）

### 2\.1 两大核心控制器

- **joint\_state\_broadcaster**：广播所有关节状态、发布关节数据

- **diff\_drive\_controller**：差速机器人专用控制器

### 2\.2 关键参数含义

- wheel\_radius：轮子半径，决定运动速度换算

- wheel\_separation：左右轮距，决定转向精度

- odom\_frame：里程计坐标系

- base\_frame\_id：机器人基坐标系

## 3\. 速度指令链路（核心必考）

Nav2/手动控制发布 /cmd\_vel → 话题转发至 /diff\_drive\_controller/cmd\_vel\_unstamped → 控制器解析速度 → 控制左右轮关节运动 → 计算并发布里程计与TF

## 4\. 控制器调试命令

```Plain Text

ros2 control list_joints
```

# 09\_差速运动学\.md

## 1\. 差速机器人运动原理

差速机器人依靠**左右驱动轮转速差**实现运动：两轮同速前进后退，转速不同实现转向，万向轮仅起支撑作用，无驱动能力。

## 2\. 核心运动学公式（通俗理解）

机器人前进速度 = 两轮平均速度；机器人旋转角速度 = 两轮速度差 / 轮距

这也是为什么 **轮距、轮半径** 参数直接影响机器人运动精度、里程计精度、导航轨迹的核心原因。

## 3\. 参数对项目的影响

- 轮半径偏大：实际运动距离大于指令距离

- 轮距不准：机器人转向偏移、导航轨迹跑偏

## 4\. 仿真与实车差异

仿真运动学完美无误差，实车存在打滑、摩擦不均，需要后续参数校准与滤波优化。

# 13\_Nav2导航\.md

## 1\. Nav2整体架构

Nav2是ROS2官方导航框架，实现机器人**定位、全局路径规划、局部避障、速度控制、故障恢复**全套自主导航功能。

## 2\. 核心模块详解

- **AMCL**：粒子滤波定位，在已知地图中确定机器人位置

- **全局规划器**：规划从起点到终点的最优全局路径

- **DWB局部规划器**：实时生成速度指令，避障、微调轨迹

- **代价地图Costmap**：将障碍物、膨胀区转化为导航可识别代价

- **速度平滑器**：限制加减速，保证机器人运动平稳

## 3\. 导航完整TF链路

map（全局地图）→ odom（里程计）→ base\_footprint（机器人基准）

## 4\. 导航调参核心

机器人半径、障碍物膨胀半径、最大速度、加速度、规划器权重，适配自身机器人尺寸与运动特性。

## 1\. 项目整体定位

本项目为**配置型系统集成项目**，无自定义C\+\+/Python源码，核心能力全部依托ROS2官方生态，通过模型、插件、参数、启动文件组合实现完整机器人功能。

## 2\. 项目核心逻辑闭环

模块化建模 → 仿真硬件模拟 → 标准化控制 → 传感器数据输出 → SLAM建图 → 地图复用 → Nav2自主导航

## 3\. 项目优势与拓展性

结构清晰、参数化程度高、适配仿真与实车、可快速拓展视觉识别、多机导航、路径跟踪等功能。

# 17\_实车迁移\.md

## 1\. 仿真与实车核心区别

仿真：数据完美、无误差、无硬件延迟；实车：存在打滑、误差、通信延迟、硬件噪声，需要驱动适配、参数校准、滤波优化。

## 2\. 模块迁移对应关系

- 仿真雷达插件 → 真实雷达驱动节点

- 仿真diff\_drive控制器 → 真实底盘通信\+电机控制

- 仿真里程计 → 编码器\+IMU融合里程计

## 3\. 迁移核心要点

模型参数、导航规划参数可大部分复用；运动学参数、噪声参数、控制延时必须实车标定。

# 18\_机器人底盘通信\.md

## 1\. 实车底盘数据链路

STM32底盘（编码器采集\+电机控制）→ 串口通信 → ROS2底盘通信节点 → 发布里程计、TF → 供给SLAM/Nav2使用

## 2\. 核心通信要点

串口波特率匹配、数据解析协议、指令限速、急停安全逻辑、数据滤波去噪。

## 3\. 与仿真的差异

仿真无通信延迟，实车必须处理数据丢包、延时、异常数据，保证导航稳定。

# 19\_常见Bug\.md

## 1\. 模型类BUG

机器人浮空、下陷、轮子不转、传感器无数据：模型joint名称不匹配、惯量参数错误、插件未加载

## 2\. 控制类BUG

机器人不运动、运动跑偏、里程计不更新：话题未转发、控制器未激活、轮参数不匹配

## 3\. SLAM/导航BUG

建图重影、定位飘、导航不动、避障失效：TF断连、传感器数据异常、代价地图参数不合理

## 4\. 编译启动BUG

修改文件不生效：未执行colcon build、未source环境变量
